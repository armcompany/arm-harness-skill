# SDD Pattern Inside Harness Engineering

Use this reference when a harness should externalize intent before code is generated. SDD here means specification-driven development: a workflow that turns product intent into explicit specs, plans, tasks, acceptance criteria, and verification sensors before implementation.

SDD is not separate from harness engineering. It is a feedforward pattern inside the harness, especially useful for behavior harnesses and long-horizon coding work.

## Why SDD Belongs In The Harness

Coding agents often fail because the desired behavior is implicit, underspecified, or changing while they work. SDD externalizes that desired behavior into durable artifacts the agent can repeatedly inspect and validate against.

Mapping to harness concepts:

- Spec: feedforward guide for desired behavior.
- Plan: control-flow guide for sequencing work.
- Tasks: scoped execution units for the agent loop.
- Acceptance criteria: termination and evaluator contract.
- Fixtures/examples: behavior sensors with human-approved ground truth.
- Tests and reviews: feedback sensors.
- Change log: memory of decisions and scope changes.

## When To Use SDD

Use SDD when:

- The task has user-visible behavior.
- Requirements are ambiguous or multi-step.
- The agent will work with high autonomy.
- Tests may be generated by the same agent and therefore need stronger grounding.
- Multiple agents or sessions will coordinate.
- A human wants lower review toil but still needs confidence.

Do not force SDD for tiny mechanical edits, obvious refactors, formatting, or one-line fixes unless the risk is high.

## SDD Workflow

1. Capture intent.
   - Restate the problem, users, constraints, non-goals, and unknowns.
   - Ask only for missing information that blocks a meaningful spec.

2. Write the spec.
   - Define expected behavior, edge cases, data contracts, UI/API states, and failure handling.
   - Prefer examples over abstract prose.
   - Mark assumptions explicitly.

3. Define acceptance criteria.
   - Make "done" observable.
   - Separate required behavior from nice-to-have behavior.
   - Include negative cases where mistakes are likely.

4. Create or identify approved fixtures.
   - Use human-provided examples, golden files, snapshots, request/response pairs, screenshots, or domain fixtures where possible.
   - Treat AI-generated tests as useful but insufficient unless reviewed or anchored to approved fixtures.

5. Produce the plan.
   - Decompose into small tasks.
   - Identify files likely in scope.
   - Identify sensors to run after each step and before finalization.

6. Implement against the plan.
   - Keep spec changes explicit.
   - Update tasks as work progresses.
   - Do not silently expand scope.

7. Verify.
   - Run computational sensors first.
   - Run inferential review only for semantic or architectural judgment.
   - Compare final behavior to acceptance criteria and fixtures.

8. Ratchet lessons.
   - If the agent misunderstood the spec, update the SDD template or guide.
   - If a failure escaped tests, add a sensor or approved fixture.

## Artifact Set

Use the smallest artifact set that matches task risk.

Lightweight:

```markdown
## Spec

## Acceptance Criteria

## Plan

## Verification
```

Full:

```markdown
# Specification

## Problem

## Users / Actors

## Desired Behavior

## Non-goals

## Constraints

## Examples / Fixtures

## Edge Cases

## Acceptance Criteria

## Implementation Plan

## Task List

## Verification Plan

## Open Questions
```

## Behavior Harness Pattern

For functional behavior, pair SDD artifacts with sensors:

| SDD artifact | Harness role | Sensor |
| --- | --- | --- |
| Functional spec | Feedforward behavior guide | Human review, evaluator review |
| Acceptance criteria | Completion contract | Checklist, PR template, finalization hook |
| Approved fixtures | Ground truth examples | Unit/integration/golden tests |
| Edge cases | Failure anticipation | Negative tests, property tests |
| Plan and tasks | Control flow | Progress file, task board |
| Verification plan | Feedback routing | CI, browser checks, logs, review agent |

## Approved Fixtures

Approved fixtures are examples trusted more than AI-generated tests because a human, production trace, contract, or external system supplied or approved them.

Examples:

- API request/response pairs.
- CSV input and expected transformed output.
- Screenshots or visual baselines.
- Golden files.
- Domain-specific examples from support tickets.
- Contract tests from another service.
- Product acceptance examples.

Use approved fixtures selectively. They are most valuable where behavior correctness matters and the expected output can be made concrete.

## SDD And Agent Autonomy

Use the SDD artifacts as permission boundaries:

- The agent can implement tasks inside the spec without asking.
- The agent must ask before changing non-goals, acceptance criteria, public contracts, or approved fixtures.
- The agent must record assumptions if work proceeds without complete answers.
- The agent must not mark the task complete until the verification plan has been executed or explicitly waived.

## Good SDD Prompt

```markdown
Use $harness-engineering to create an SDD harness for this feature before implementation. Produce a compact spec, acceptance criteria, approved fixture candidates, implementation plan, and verification plan. Then implement only after the spec is coherent.
```

## Anti-Patterns

- Writing a long spec that does not produce concrete acceptance criteria.
- Letting the implementation agent freely rewrite the spec to match its code.
- Treating AI-generated tests as proof of behavior without approved examples.
- Keeping the plan in chat only, where it disappears across context resets.
- Running semantic review before cheap deterministic checks.
- Applying full SDD to trivial changes and slowing the loop unnecessarily.
