# Harness Control Taxonomy

Use this reference when classifying, auditing, or designing coding-agent harness controls.

## Core Model

`Agent = model + harness`.

The harness is everything around the model: prompts, project rules, skills, tool descriptions, shell access, filesystem and git state, MCP servers, sandboxes, hooks, CI, planning loops, context management, memory, observability, subagents, permission gates, and recovery paths.

Do not treat harness engineering as adding more instructions. Treat it as designing a control system that steers an agent toward a desired state and gives it useful feedback when it drifts.

## Direction

| Direction | Purpose | Examples | Failure mode if alone |
| --- | --- | --- | --- |
| Feedforward guide | Steer before action | AGENTS.md, CLAUDE.md, skills, architecture docs, LSP/code intelligence, MCP docs, examples, sprint contracts | The agent may never learn whether the guide worked |
| Feedback sensor | Observe after action and support correction | tests, type checks, linters, static analysis, browser screenshots, logs, review agents, CI, mutation testing | The agent may keep repeating preventable mistakes |

Good harnesses combine both. Add guides to reduce first-pass error probability, and sensors to expose remaining errors early.

## Execution Type

| Type | Strength | Cost | Examples |
| --- | --- | --- | --- |
| Computational | Deterministic, cheap, reliable | Milliseconds to minutes | tests, type checks, lint, dependency rules, Semgrep, ArchUnit, dep-cruiser, codemods, grep checks |
| Inferential | Semantic judgment, richer context | Slower, probabilistic | AI code review, architecture review agents, LLM judges, product-quality evaluators |

Use computational controls whenever the desired property can be expressed structurally. Use inferential controls for semantic duplication, overengineering, misunderstood intent, architectural tradeoffs, and quality judgment.

## Regulation Categories

| Category | What it regulates | Typical guides | Typical sensors |
| --- | --- | --- | --- |
| Maintainability | Internal code quality and review toil | coding conventions, refactoring rules, test patterns | lint, type checks, complexity, duplication, coverage, review agents |
| Architecture fitness | Architectural characteristics and boundaries | architecture docs, topology templates, observability standards | module-boundary tests, dependency rules, performance tests, API contract checks |
| Behavior | Functional correctness from a user's perspective | specs, examples, approved fixtures, acceptance criteria | tests, browser checks, golden fixtures, mutation testing, manual QA |
| Security and permissions | Harm prevention and least privilege | threat model, safe command policy, MCP allowlist | secret scanners, SAST, destructive-command hooks, permission gates |
| Runtime reliability | Production health and operations | SLOs, logging conventions, debugging playbooks | error-rate alerts, latency dashboards, log anomaly judges, trace sampling |

Behavior harnessing is usually the hardest category because tests generated by the same agent are not enough. Prefer human-approved fixtures, externally supplied examples, contract tests, and production-derived signals where possible.

## Lifecycle Placement

Place controls as far left as their speed and reliability allow.

| Stage | Good controls |
| --- | --- |
| Before generation | AGENTS.md, skills, architecture docs, LSP, examples, service topology templates |
| Agent self-correction loop | fast tests, lint, type checks, browser screenshots, local logs, basic review agent |
| Pre-commit or pre-push | relevant test subset, formatting, secret scan, destructive-pattern checks |
| Pull request | broader tests, semantic review, architecture review, coverage and mutation sampling |
| Post-integration | full CI, drift scans, dependency checks, dead-code detection |
| Continuous runtime | SLO monitors, log anomaly detection, response-quality sampling, production feedback mining |

## Harnessability

A codebase is easier to harness when it has ambient affordances:

- Strong types or schemas.
- Clear module boundaries.
- Stable service topology.
- Deterministic tests and local bootstrap.
- Machine-readable architecture rules.
- Good logs and traces.
- Small, conventional workflows.

For low-harnessability codebases, improve structure and sensors first. Do not try to compensate with long prompts.
